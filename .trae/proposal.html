创意提案

AI 创作工作台 —— 把角色、场景、分镜结构化管理起来的 AI 视频素材生成器。

一个本地优先的桌面应用，帮你在本地数据库里管理角色设定、场景资料、分镜编排和元素绑定，通过统一接口调用 13 家 AI 提供商，逐镜生成视频素材，最后导出可用于剪辑软件的素材包。

01 · 为什么做这个

遇到三个真实的麻烦

做系列视频内容的创作者，大概都会碰到以下这些问题。

提示词和参考图要反复写、反复传
同一个角色、同一个场景，在每个分镜的提示词里都要重写一遍描述，还要反复上传同一张参考图。改一次角色设定，意味着回头改十几个已经做好的分镜。
素材散落在各家平台，版本混乱
你在 A 平台生成的视频、B 平台生成的角色图、C 平台生成的配音，文件名随意，版本无记录。重做一个镜头时，之前的参考图可能已经找不到了。
模型差异让人头大
有的模型支持首末帧，有的不支持；有的需要角色图，有的不需要；有的要竖版参数，有的要横版。切换模型意味着重新了解一套体系。
数据在云端，控制权不在你手上
你辛苦搭建的角色设定、场景资料、分镜规划，全都保存在各个平台的服务器上。账号封停、平台关闭，数据就可能一起没了。
02 · 面向谁

给谁做的

系列内容创作者
想做系列视频内容，但没有经验从零搭建自己的工作流，也没有预算在多家平台反复试错。
UP 主与独立创作者
有持续产出角色 / 场景的需要，希望把这些信息系统化管理，而不是散落在笔记里。
重视数据所有权的人
不想把作品和角色数据留在第三方平台，希望所有创作信息都在本地。
想快速验证创意的团队
有了故事想法，希望用最短时间把它变成可视化的素材包，用于评估和后续剪辑。
03 · 它做什么

核心功能与使用流程

典型的 5 步使用流程
STEP 1
创建项目
在本地数据库创建一个新项目，所有数据独立保存。
STEP 2
录入角色与场景
角色属性、服装、参考图只需录入一次，可复用。
STEP 3
编排分镜并绑定元素
每个分镜绑定对应的角色和场景，系统级关联。
STEP 4
自动调用 AI 生成
系统拼装提示词，统一调用 13 家 AI 提供商，逐镜生成视频素材，版本可回退。
STEP 5
导出素材包
把项目整体导出，包含视频、参考图、元数据，直接导入剪辑软件。
5 个核心能力
  1. 结构化内容管理
    角色不是提示词里的文字描述，而是可复用的资料卡 — 包含基本属性、多套服装与表情、参考图管理、特征锚定信息。场景同理，不是临时输入的文字，而是可跨项目复用的场景档案。分镜不是一行一行的提示，是绑定了角色和场景的结构化节点，提示词由系统自动拼装，不需要手动复制。角色和场景在分镜间保持一致，系统能比对前后生成结果并提醒角色外观漂移。
  2. 视频素材生产与管理
    逐镜生成视频素材，每个分镜独立调用 AI 提供商。任务状态机覆盖待提交 / 已提交 / 生成中 / 已完成 / 失败 / 已取消的完整流程。失败可自动重试、可恢复到上次失败的那一镜继续，不需要重头生成。本地文件缓存避免重复调用 AI。对同一个分镜可以保存多个生成版本，可一键切换、对比、回退。
  3. 多模型统一接入与插件系统
    内置 13 家主流 AI 提供商（快手可灵 Kling / Runway / Pika / Luma AI / Seedance / MiniMax / 智谱 AI / OpenAI & Sora / Anthropic Claude / Google Vertex / 火山引擎 / PixVerse / OpenAI 兼容）。两种插件方式：声明式 JSON 插件不加代码只写配置就能接入新提供商；沙箱隔离的代码插件运行在 vm 沙箱中，无权访问本地文件、不能发起内网请求，有 URL 白名单和 SSRF 防护。不管哪一家 AI，用户看到的都是同一套输入方式（提示词、时长、分辨率、参考图、运动强度、种子数等），系统内部做提供商间的参数翻译。每次 AI 调用都记录请求/响应、耗时、费用、错误信息，可配置每家的调用频率和每日/每月限额，发布新版本模型时能平滑切换。
  4. 素材与版本管理
    逐镜导出视频文件 + 参考图 + 元数据 JSON，一键打包为剪辑软件可直接打开的文件夹结构。对多个分镜的生成结果做批量重命名、批量导出、批量清理。整个项目可打包迁移到另一台机器，包括所有角色、场景、分镜、素材、设置。
  5. Agent 对话助手
    告诉 Agent 你想用哪家 AI，它帮你配置好接口。生成失败时，Agent 分析错误信息并给出参数调整建议。根据你写的情节结构，主动提出分镜建议。

04 · 价值与意义

它能带来什么

  1. 效率提升
    把角色 / 场景 / 分镜从提示词里拎出来，变成结构化数据。提示词由系统自动拼装，单分镜的生成准备时间大幅缩短。角色修改后，所有引用该角色的分镜自动同步，不需要手动回去改。
  2. 降低创作门槛
    不需要了解各家 AI 模型的参数体系，不需要记住每个模型的差异，也不需要写代码。统一的界面调用主流提供商，专注内容而不是工具。
  3. 数据主权
    所有角色、场景、分镜、素材记录存在本地 SQLite 数据库；API 密钥使用系统级加密存储（Windows Credential Manager / macOS Keychain）；项目可整体导出、备份、迁移。不需要云账号，不依赖任何平台。

架构设计（DDD 六层架构）
  • app 层 —— 页面组件和布局，是最终用户能看到的界面。
  • modules 层 —— 9 个业务模块（story、video、shot、character、scene、prompt、asset、sync、persistence），每个模块有明确定义的 public API 和不变量。
  • domain 层 —— 纯类型层，零外部依赖。定义所有业务实体的 Zod schemas 和 Port 接口。
  • shared-logic 层 —— 零外部依赖的纯逻辑层（提示词构建、一致性判断、视频参数、引用匹配）。可在主进程和渲染进程双向复用。
  • shared 层 —— 跨模块通用的 UI 组件、工具函数、常量。
  • infrastructure 层 —— 基础设施（DI 容器、存储、AI 提供商适配、网络防护）。

依赖方向由 ESLint 规则强制约束，跨模块由契约定义。修改一个模块时，系统会验证是否违反其他模块的不变量。

06 · 怎么来的

这个项目的由来

  • 最开始想做 AI 视频，但市面上的工具都太难上手了 —— 提示词重复写、素材散落在各家平台、换模型要重新学一套。那时候正好在研究 OpenClaw，就用它搭了一个最初的原型版本。
  • 做着做着发现这个方向值得做完整一点，但 OpenClaw 的成本和打包后的白屏问题一直卡着解决不了，决定换方案。
  • 转到 Trae Solo 后，之前卡住的白屏问题直接通了；之后就是在 Trae Solo 中多轮迭代、优化和重构，逐步形成现在的版本。
  • 项目核心代码在 Trae Solo 中完成。从数据库 Schema 到各业务模块的服务、hook、组件，再到 13 家 AI 提供商的适配 —— 大部分实现都由 Trae 生成初版后再由人来调试和完善。
对 AI 辅助开发的一点体会

最开始让 Trae 扫 bug，每次扫出来的都是一大堆；修完后再扫，又是新的一大堆。反复几轮之后意识到，这大概不是 bug 多的问题，是代码结构本身在阻碍改动——每修一个地方，好几个地方跟着崩。

于是让 Trae 做了一次高度模块化重构。重构完之后问 AI：从 AI 开发的角度看，这个架构有没有问题？AI 分析了当前结构，提出了 DDD 分层 + 子域化模块的方案。这个方案的核心逻辑是：每一层有明确的依赖方向，每个模块有自己的 public API 和不变量——这样 AI 改代码时不容易跨层误伤，出问题也能快速定位。

后续又陆续加了 contract.json（让 AI 每次改模块前先读契约）、回归防护规则库（修过的 bug 不犯第二次）、架构校验脚本（改错时立刻报错）。这些都不是一次性设计好的，是一轮一轮迭代长出来的。

最大的感受是：当大部分代码由 AI 辅助完成时，系统的可维护性不只取决于"人能不能看懂"，也取决于"AI 能不能在改动时做出正确的最小判断"。这也是目前还在继续摸索的方向。
