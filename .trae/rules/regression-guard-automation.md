# Regression Guard Automation Protocol

> 本文档定义 AI 在发现 bug 后的自动化处理流程：
> 判断是否加入回归防护 → 编写回归测试 → 整理防护规则

---

## 一、触发条件

当以下任一情况发生时，AI 自动启动本流程：

1. **修复了一个 bug** — 无论谁发现的，只要代码被修改来修复问题
2. **发现了一个潜在 bug** — 在代码审查或功能开发中发现的问题模式
3. **测试失败** — 现有测试暴露了回归问题
4. **用户报告了问题** — 通过 issue 或反馈渠道

---

## 二、决策框架：是否加入回归防护？

AI 发现 bug 后，必须回答以下 5 个问题。**全部为"是"才加入回归防护**：

### Q1: 可复现性 — 这个 bug 是否有明确的触发条件？

- **是** — 存在具体的操作序列或代码路径可以稳定复现
- **否** — 仅在不可预测的条件下偶发（如网络抖动、GC 时序）→ 不加入回归防护，改为加监控日志

### Q2: 回归风险 — 修复后，同类问题是否可能被重新引入？

- **是** — 修复不是结构性的（如改了一个条件判断），未来修改相关代码时容易重新引入
- **否** — 修复是根本性的（如删除了一个整个有问题的模块），同类问题不可能再发生 → 不加入回归防护

### Q3: 检测可行性 — 是否能编写自动化检测（测试或 lint 规则）？

- **是** — 可以编写单元测试、集成测试、ESLint 规则或架构扫描脚本来检测
- **否** — 只能通过人工审查或运行时观察发现 → 加入 CR 规则（代码审查清单），不加入自动化防护

### Q4: 影响严重度 — 这个 bug 对用户的影响是否达到 P2 及以上？

- **是** — P0（数据丢失）、P1（功能不可用）、P2（体验退化）
- **否** — P3（轻微不便，如 UI 对齐偏差）→ 不加入回归防护，仅在 commit message 中记录

### Q5: 规则通用性 — 这个 bug 模式是否可能在其他地方出现？

- **是** — 同样的代码模式存在于多个文件或模块
- **否** — 仅在唯一一处出现，且该处已被修复 → 加入针对性单元测试，不加入通用回归防护规则

### 决策矩阵

| Q1 | Q2 | Q3 | Q4 | Q5 | 动作 |
|----|----|----|----|-----|------|
| ✓ | ✓ | ✓ | ✓ | ✓ | 加入回归防护 + 回归测试 + 通用规则 |
| ✓ | ✓ | ✓ | ✓ | ✗ | 加入回归防护 + 针对性测试 |
| ✓ | ✓ | ✗ | ✓ | ✓ | 加入 CR 规则 + 通用规则（无自动化检测） |
| ✗ | - | - | - | - | 加监控日志，不加入回归防护 |
| ✓ | ✗ | - | - | - | 不加入回归防护（修复已根除问题） |
| ✓ | ✓ | ✓ | ✗ | - | 不加入回归防护（影响太小） |

## 三、回归测试编写规范

### 3.1 测试模板

```typescript
import { describe, it, expect, vi, beforeEach } from "vitest";

describe("Regression Guard: R{N} — {规则名称}", () => {
  beforeEach(() => {
    vi.clearAllMocks();
  });

  it("should {安全行为描述}", async () => {
    // Arrange: 设置触发 bug 的前置条件
    const { result } = renderHook(() => useXxx());

    // Act: 执行触发 bug 的操作
    await act(async () => {
      await result.current.dangerousOperation();
    });

    // Assert: 验证安全模式生效
    expect(result.current.xxx).toBe(expectedValue);
  });

  it("should NOT {危险行为描述}", async () => {
    // 验证危险模式不会发生
    // 例如：验证不会出现数据丢失、不会出现静默失败等
  });
});
```

### 3.2 测试命名规则

```
Regression Guard: R{编号} — {规则名称}
```

编号从 R110 开始（当前已有 R1-R151），按顺序递增。最新规则数与分类见 [regression/index.md](regression/index.md)。

### 3.3 测试文件位置

| Bug 所在层 | 测试文件位置 |
|-----------|-------------|
| `src/modules/{module}/` | `src/modules/{module}/{subdomain}/__tests__/regression-r{n}.test.ts` |
| `src/shared-logic/` | `src/shared-logic/{sub}/__tests__/regression-r{n}.test.ts` |
| `electron/src/api/` | `electron/src/__tests__/regression-r{n}.test.ts` |
| `electron/src/services/` | `electron/src/__tests__/regression-r{n}.test.ts` |

### 3.4 测试必须包含的场景

1. **正向测试** — 验证安全模式生效
2. **反向测试** — 验证危险模式不会发生（如果可行）
3. **边界测试** — 验证在边界条件下安全模式仍然生效

## 四、回归防护规则编写规范

### 4.1 规则格式

每条规则必须包含以下字段：

```markdown
### R{N}: {规则名称}

{一句话描述规则的核心约束}

**BAD**:
\```typescript
{危险代码模式}
\```

**GOOD**:
\```typescript
{安全代码模式}
\```

**检测方法**:
- [自动] {ESLint 规则 / 架构扫描脚本 / 单元测试}
- [半自动] {代码审查检查项}

**对应场景**: {触发此规则的用户场景}

**已知例外**: {什么情况下危险模式是安全的}
```

### 4.2 规则分类

| 类别 | 编号范围 | 说明 |
|------|---------|------|
| 数据一致性 | R1-R30（及后续扩展 R141 等） | 数据不丢、不脏、不冲突 |
| 异步安全 | R31-R50（及后续扩展 R140 等） | 并发、竞态、轮询、生命周期 |
| 错误处理 | R51-R65（及后续扩展 R134、R136 等） | 错误不吞、不假成功、用户可理解 |
| UI 健壮性 | R66-R80 | 界面不崩、有反馈、无泄漏 |
| 工程质量 | R81-R100（及后续扩展 R135 等） | 依赖合规、构建安全、测试可靠 |
| 平台兼容 | R101-R110 | IPC、Electron 环境、进程模型 |
| 用户安全防护 | R111-R120 | 破坏性操作需确认、数据清除需保护 |
| 系统安全 | R121-R130（及后续扩展 R131-R133、R137-R139、R142 等） | 沙箱隔离、并发保护、资源生命周期 |

> **说明**：上表中的"编号范围"为初始规划区间，仅作分类参考。实际规则编号已扩展至 R151，
> 部分新规则编号超出原区间但归属同一分类。完整的规则编号到分类映射以
> [regression/index.md](regression/index.md) 的分类表为准（当前总计 151 条）。

### 4.3 自动化检测优先级

1. **ESLint 规则** — 最高优先级，编译时捕获
2. **架构扫描脚本** (`scripts/check-architecture.mjs`) — CI 时捕获
3. **单元测试** — 运行时捕获
4. **CR 规则** — 人工审查时捕获

## 五、自动化工作流

### 5.1 发现 bug 时的自动流程

```
AI 发现 Bug
  │
  ├── Step 1: 修复 Bug
  │     └── 记录修复内容（改了什么、为什么改）
  │
  ├── Step 2: 决策评估（回答 Q1-Q5）
  │     ├── 不需要回归防护 → 结束
  │     └── 需要回归防护 → 继续
  │
  ├── Step 3: 编写回归测试
  │     ├── 创建测试文件 regression-r{n}.test.ts
  │     ├── 包含正向 + 反向 + 边界测试
  │     └── 运行测试确认通过
  │
  ├── Step 4: 编写回归防护规则
  │     ├── 按规则格式写入 regression-guards.md
  │     ├── 分配编号 R{N}
  │     └── 标注检测方法
  │
  ├── Step 5: 实现自动化检测（如果可行）
  │     ├── ESLint 规则 → eslint.config.mjs
  │     ├── 架构扫描 → scripts/check-architecture.mjs
  │     └── 或标注为 CR 规则
  │
  └── Step 6: 更新项目文档
        ├── project_rules.md — 更新回归守卫计数
        └── MODULE.md — 如果影响模块契约，更新 invariants
```

### 5.2 定期整理流程

每完成一轮 bug 审计或功能迭代后，执行以下整理：

1. **编号连续性检查** — 确认 R1-R{N} 无缺号
2. **分类一致性检查** — 确认每条规则的分类正确
3. **检测方法有效性检查** — 确认标注为"自动"的规则确实有对应的 ESLint/脚本/测试
4. **重复规则合并** — 如果两条规则描述同一模式，合并为一条
5. **过时规则清理** — 如果规则对应的代码已完全重构（模式不再存在），标注为 `[已归档]`

## 六、与现有流程的集成

### 6.1 与 bug-audit-methodology.md 的关系

- **审计方法论** 定义如何发现 bug（阶段一、二、三）
- **本协议** 定义发现 bug 后如何自动化处理（决策、测试、规则、整理）
- 两者互补：审计方法论产出 bug 列表，本协议将 bug 转化为持久防护

### 6.2 与 project_rules.md 的关系

- `project_rules.md` 的 "Regression Guards" 摘要表必须与 `regression-guards.md` 的实际规则数同步
- 每次新增规则后，更新 `project_rules.md` 中的总数和分类计数

### 6.3 与 CI/CD 的关系

- `npm run validate:full` 必须包含回归测试的执行
- `npm run lint:arch` 必须包含架构扫描中的回归规则检测
- Pre-commit hook 必须运行受影响模块的回归测试

## 七、快速参考

### AI 发现 Bug 后的检查清单

```
□ 修复 Bug
□ Q1: 可复现？ → 否 = 加监控，结束
□ Q2: 会回归？ → 否 = 结束
□ Q3: 可自动检测？ → 否 = 加 CR 规则，跳过 Step 5
□ Q4: 严重度 ≥ P2？ → 否 = 记录，结束
□ Q5: 通用模式？ → 是 = 通用规则；否 = 针对性测试
□ 编写回归测试 (regression-r{n}.test.ts)
□ 编写回归防护规则 (regression-guards.md)
□ 实现自动化检测 (ESLint/脚本)
□ 更新 project_rules.md 计数
```
