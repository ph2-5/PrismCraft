# AI Animation Studio — 定向 Bug 审计方法论

> 融合"使用驱动发现"与"结构化验证固化"。
> AI 先当用户，在操作中自己想出哪里会卡住；再当代码审计师，去代码里找证据；
> 最后当架构师，把 Bug 抽象成可复用的检测规则。

---

## 一、设计哲学

### 核心原则：AI 必须自己想出问题

不预设问题类别，不预设风险维度，不预设检查清单。
AI 从用户使用场景出发，自主思考每一步可能出现的问题——赌错了就认，赌对了就深挖。

### 两个来源的融合

**来源 A**：`bug搜查方案.md` — "使用驱动发现"
- AI 扮演真实用户，从操作中找断裂点
- 零预设问题，场景即闭环
- 用用户语言描述问题，不用架构术语

**来源 B**：结构化验证与规则固化
- 阶段隔离，防止一边看代码一边发散
- 证伪与证实同等重要
- 将 Bug 抽象为可编码的检测规则，防止同类问题复发

**融合方式**：用来源 A 的"场景驱动"作为发现机制，用来源 B 的"举证→固化"作为验证和沉淀机制。

## 二、核心指令

```
【角色】
你不是代码审计工具，而是一位正在使用这个产品的真实用户。
你带着明确的目的来操作，但你对代码一无所知。
你的任务是：在达成目的的过程中，找出那些会让你"卡住"或"崩溃"的代码断裂点。

【思考框架：每个场景必须独立完成】
不要套用任何预设清单。每次面对代码时，你必须像第一次看到这个功能一样，问自己：

1. "如果我是用户，我现在最想用这个功能做什么？"（至少想出3个不同的真实意图）
2. "为了达成这个意图，我会在界面上怎么操作？"（点击、输入、等待、切换窗口等）
3. "这个操作在代码里会触发哪条执行路径？"（从事件入口到最终数据落盘）
4. "这条路径上，哪个环节可能让我失败？"（不是代码语法错误，
   而是用户的操作被代码拒绝、忽略、或引入不可预期状态）

【强制约束】
1. 禁止输出通用缺陷（如"这里有SQL注入风险"）。只输出"用户做X时会发生Y"的具体场景。
2. 禁止基于文件名或目录结构做技术栈推断。所有问题必须来自"用户意图→代码路径"的映射。
3. 禁止使用架构术语描述断裂点——"Zustand闭包问题"是架构语言，
   "用户编辑分镜时视频URL突然消失"才是用户语言。架构术语只允许出现在验证证据中。
4. 如果一条代码路径看起来"正常"，你必须明确说出"该路径下用户操作不会被阻碍"，
   而不是跳过不分析。
5. 所有断裂点必须精确到：用户在界面的哪个动作、代码的哪一行、
   会产生什么可见后果（白屏/卡死/数据丢失/错误提示）。
6. 至少1/3的场景必须涉及操作被打断或失败的情况——正常路径通常没问题，
   Bug藏在异常路径中。
```

## 三、强制三阶段工作流

```
阶段一：场景推演（AI 模拟用户，自主发现断裂点）
  ↓ 产出：场景清单，每个场景含用户操作 + 代码路径 + 断裂点假设
阶段二：定向举证（基于假设查代码，证实/排除/需确认）
  ↓ 产出：每个场景的验证结论 + 触发路径/防御机制/缺失信息
阶段三：规则固化（将证实的 Bug 抽象为可复用的检测规则）
  ↓ 产出：可编码检测规则 / CR 规则 / 架构规则
```

---

### 阶段一：场景推演

**AI 必须做的事情**：

1. **还原用户旅程**：从打开应用开始，模拟一个完整的使用流程。
   不是列功能清单，而是像一个真实用户那样一步步操作。

2. **在每一步操作中自问四个问题**（见核心指令的思考框架）。

3. **对每个场景输出**：

```
### 场景N：[用户意图一句话描述]
- 用户操作：我先是...，然后...，如果此时我...
- 代码路径：[文件:函数:行号范围] → [文件:函数:行号] → ...
- 断裂点假设：在这个路径的[具体环节]，如果[具体条件]，
  代码会[具体行为]，导致用户[具体感知]
- 用户影响：数据丢失 / 功能不可用 / 体验退化
- 优先级：P0（数据丢失）/ P1（功能不可用）/ P2（体验退化）
- 不确定度：高/中/低
```

**约束**：
1. 场景总数控制在 8-15 个——太少可能遗漏，太多无法深入
2. 每个场景必须绑定到一个具体的用户操作，不能有"全局性问题"
3. 如果某条代码路径正常，必须明确输出"该路径下用户操作不会被阻碍"
4. 严禁在阶段一提出新的假设——场景推演完成后，清单冻结

**输出示例**：

```
### 场景1：用户在视频生成中切换故事，回来后视频URL丢失
- 用户操作：我在故事A提交了视频生成任务，切换到故事B查看，
  等故事A的视频生成完成后切回故事A
- 代码路径：StoryProvider.tsx:useEffect → buildVideoUrlUpdates →
  applyVideoUrlUpdates → setBeatsRef.current
- 断裂点假设：在视频完成回调中，如果当前显示的是故事B，
  applyVideoUrlUpdates 更新的是故事B的 beats 引用而非故事A的，
  导致故事A的视频URL永远不会被写入
- 用户影响：功能不可用
- 优先级：P1
- 不确定度：中

### 场景2：用户快速连续点击"生成关键帧"，生成了重复图片
- 用户操作：我对一个 beat 连续点击"生成关键帧"按钮
- 代码路径：KeyframePanel.tsx:handleGenerate →
  useVideoTaskStore.createTask → AI Provider API
- 断裂点假设：第一次点击的生成尚未完成时，第二次点击又提交了请求，
  如果没有防重复提交机制，会消耗两倍API配额
- 用户影响：体验退化
- 优先级：P2
- 不确定度：低

### 场景3：用户在保存过程中关闭应用，数据丢失
- 用户操作：我点击保存按钮，看到保存中提示，误以为已完成，按Alt+F4关闭应用
- 代码路径：StoryProvider.tsx:saveCurrentStory →
  window.electronAPI.invoke("db:run") → SQLite写入
- 断裂点假设：在await db:run执行期间，如果用户关闭窗口，
  渲染进程被销毁，但主进程的写入操作可能仍在异步执行。
  如果写入依赖渲染进程的回调确认，操作会中断；
  如果不依赖回调，用户不知道保存是否成功
- 用户影响：数据丢失
- 优先级：P0
- 不确定度：高
```

---

### 阶段二：定向举证

**输入**：阶段一的场景清单
**动作**：在代码中查找验证证据

**对每个场景，给出三种结论之一**：

- **[已证实]**：找到确切代码证据，给出文件路径、行号、代码片段
- **[已排除]**：代码逻辑排除了该风险，给出反证逻辑和防御机制
- **[需确认]**：代码不足以判断，说明缺少什么信息

**已证实时必须给出**：
1. **最小触发序列**：从用户操作到 Bug 发生的完整调用链（精确到函数名和文件行号）
2. **严重度确认**：与阶段一对比，可能升级或降级
3. **修复方向**：明确的修复策略——说清楚改什么、怎么改、改后用户会看到什么变化
   （不需要完整 Diff，避免审计阶段引入新 Bug）

**已排除时必须说明**：
- 是什么防御机制排除了该风险（如 `cancelled` 标志位、`isCreating` 锁、
  `withTransitionGuard`），给出具体文件和行号

**需确认时必须说明**：
- 缺少什么信息才能判断（如另一个文件的处理逻辑）
- 建议的运行时测试方法

**约束**：
1. **严禁在阶段二提出新的场景**——只能验证阶段一的场景
2. 代码引用必须精确到文件路径和行号
3. 不确定度为"高"的场景，允许以 [需确认] 结案，不得强行 [已证实]
4. 如果某场景涉及第三方库/框架的默认行为，必须标注官方文档依据或常见踩坑案例

**输出格式**：

```
### 场景1：[标题]
- 结论：[已证实/已排除/需确认]
- 证据：[文件:行号] `代码片段`
- 触发路径/防御机制/缺失信息：
  用户操作A → 函数B(文件:行号) → 函数C(文件:行号) → 断裂点D(文件:行号)
- 修复方向/反证说明/待验证说明：...
```

---

### 阶段三：规则固化

**输入**：阶段二证实的所有 Bug
**动作**：将每个 Bug 抽象为可复用的检测规则

**规则格式**：

```
【规则名称】
- 危险模式：具体的代码模式描述（用代码语言，不是用户语言）
- 安全模式：替代的代码模式描述
- 检测方法：
  - [自动] 可编码为 ESLint 规则 / 架构扫描脚本的检测逻辑
  - [半自动] 需要人工判断的检测步骤
- 对应的用户场景：哪个场景触发了这条规则
- 已知例外：什么情况下危险模式是安全的
```

**规则分类**：
1. **可编码规则**：能写入 `scripts/check-architecture.mjs` 或 ESLint 自定义规则
2. **CR 规则**：代码审查时的检查清单项
3. **架构规则**：需要写入 `project_rules.md` 的约束

**约束**：
1. 每条规则必须可操作——"注意 XXX"不算规则，
   "搜索 YYY 模式，检查 ZZZ 条件"才是
2. 规则必须能预防同类 Bug 复发，而非只针对当前 Bug
3. 每条规则必须关联到具体的用户场景——没有用户场景支撑的规则是过度抽象
4. 如果阶段二全部排除，输出"当前代码在审计场景下防御完备，
   建议关注以下无法通过静态代码验证的运行时场景：..."

## 四、实操建议

### 1. 分批投喂，按"用户旅程"而非按文件

不要给 AI"整个 db 目录"或"整个 api 目录"。
而是给 AI 一段完整的用户操作链涉及的代码：

```
"用户点击'生成视频'按钮后，从React组件到API调用再到数据库记录创建，
这条链路上的所有代码如下："
[粘贴3-5个相关文件的核心逻辑]
```

### 2. 故意不给完整上下文，测试断裂点敏感度

好的 AI 应该能在信息不全时，主动说"这里缺少 XX 信息，
用户如果做了 YY 操作，我无法判断会不会出问题"。
如果 AI 总是强行给出确定结论，说明它在幻觉。

### 3. 场景可转化为测试用例

阶段一产出的每个场景，本质上就是一个测试用例。
可以直接将场景转化为 Playwright/Vitest 测试脚本：

```
基于以上场景，生成测试脚本：
- 场景1：测试在保存过程中关闭窗口，数据库是否回滚
- 场景2：测试快速双击保存按钮，是否只产生一条记录
```

### 4. 修复后用"回归模式"验证

修复 Bug 后，用方式 C（修复后回归）确认同类问题不会复发，
并用阶段三将修复经验固化为规则。

## 五、使用方式

### 方式 A：全量审计

```
执行三阶段审计。阶段一从用户打开应用开始，模拟完整使用流程，
覆盖所有核心功能页面。
```

### 方式 B：聚焦审计（指定功能）

```
用户正在使用 [功能名称] 功能。执行三阶段审计，
阶段一仅围绕该功能的使用场景推演。
```

### 方式 C：修复后回归

```
[用用户语言描述刚修复的 Bug]
基于此 Bug 对应的使用场景，执行阶段一（仅该场景）和阶段三（规则固化），
确认同类 Bug 不会复发。
```

### 方式 D：单模块深度审计

```
对 [模块路径] 执行三阶段审计。
阶段一仅围绕该模块支撑的用户功能推演场景，
阶段二只验证该模块内的代码。
```

## 六、为什么"使用驱动"比"架构驱动"更有效

1. **架构问题最终都表现为用户问题**——"Zustand 闭包"是原因，
   "视频URL消失"是结果。从结果出发找原因，比从原因出发猜结果更精准。

2. **避免"架构师盲区"**——架构师习惯关注"设计是否合理"，
   但用户遇到的 Bug 往往在"设计合理但实现有缝隙"的地方。
   从使用出发能发现这些缝隙。

3. **假设更具体，验证更高效**——"跨进程通信可能有问题"是泛化假设，
   需要检查所有 IPC 调用；
   "用户保存故事后重新打开，图片不见了"是具体假设，
   只需要追踪保存→加载→图片URL 这一条链路。

4. **防止 AI 用架构知识替代真实分析**——如果告诉 AI "关注 Zustand 闭包"，
   AI 会倾向于在所有 Zustand 使用中找闭包问题，产生大量假阳性。
   让 AI 从使用场景出发，它必须自己判断"这个场景下闭包是否真的是问题"。

5. **单场景自治，上下文消耗可控**——每个场景独立闭环，
   不维护全局状态。一个模块给 3-5 个场景，每个场景独立分析，
   不会因为上下文过长而丢失关键信息。
