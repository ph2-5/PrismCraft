从使用角度提问，本质上是让AI从"代码审计师"变成"用户旅程测试员"——不找代码哪里写错了，而是找用户想干某件事时，代码会不会让他失败。这个思路是对的，而且完全规避了之前所有方案的缺陷。

---

核心指令（零预设问题，纯使用驱动）

```
【角色】
你不是代码审计工具，而是一位正在使用这个产品的真实用户。你带着明确的目的来操作，但你对代码一无所知。你的任务是：在达成目的的过程中，找出那些会让你"卡住"或"崩溃"的代码断裂点。

【思考框架：每次审计必须独立完成】
不要套用任何预设清单。每次面对代码时，你必须像第一次看到这个功能一样，问自己：

1. "如果我是用户，我现在最想用这个功能做什么？"（至少想出3个不同的真实意图）
2. "为了达成这个意图，我会在界面上怎么操作？"（点击、输入、等待、切换窗口等）
3. "这个操作在代码里会触发哪条执行路径？"（从事件入口到最终数据落盘）
4. "这条路径上，哪个环节可能让我失败？"（不是代码语法错误，而是用户的操作被代码拒绝、忽略、或引入不可预期状态）

【输出要求】
对每个你想出的用户使用场景，按以下格式输出：

### 场景N：[用户意图一句话描述]
- 用户操作：我先是...，然后...，如果此时我...
- 代码路径：[文件:函数:行号范围] → [文件:函数:行号] → ...
- 断裂点假设：在这个路径的[具体环节]，如果[具体条件]，代码会[具体行为]，导致用户[具体感知]
- 验证结果：[已证实/已排除/需确认]
  - 若已证实：给出触发该断裂点的最小操作序列 + 修复代码Diff
  - 若已排除：给出代码中的防御机制 + 具体行号
  - 若需确认：说明需要补充什么信息（如另一个文件的处理逻辑）才能判断

【强制约束】
1. 禁止输出通用缺陷（如"这里有SQL注入风险"）。只输出"用户做X时会发生Y"的具体场景。
2. 禁止基于文件名或目录结构做技术栈推断。所有问题必须来自"用户意图→代码路径"的映射。
3. 每次审计的场景必须不同。如果上次审计过"用户快速双击保存按钮"，下次必须想出新场景（如"用户在保存过程中拔掉网线"）。
4. 如果一条代码路径看起来"正常"，你必须明确说出"该路径下用户操作不会被阻碍"，而不是跳过不分析。
5. 所有断裂点必须精确到：用户在界面的哪个动作、代码的哪一行、会产生什么可见后果（白屏/卡死/数据丢失/错误提示）。
```

---

为什么这个方案没有之前的缺陷

之前缺陷	本方案如何规避	
阶段一幻觉推断	没有阶段一。AI不看文件列表，直接基于代码内容想象用户怎么用	
禁止新假设导致漏报	无预设假设，每个场景都是AI现场创造的，不存在"新/旧"假设之分	
通用指令抹平技术栈	不依赖技术栈知识。Electron的IPC问题会被描述为"用户点击按钮后窗口没反应"，SQLite问题会被描述为"用户输入保存后数据没了"——从用户感知出发，技术栈差异被自然覆盖	
修复Diff危险	因为场景精确到"用户操作→代码路径"，AI必须说明"修改后用户会看到什么变化"，天然附带影响面	
上下文消耗过大	单场景自治。一个模块给3个场景，每个场景独立闭环，不维护全局状态

一个示例（展示AI会如何自主思考）

你给AI的代码：

```javascript
// src/components/ProjectSaveButton.jsx
function ProjectSaveButton({ project }) {
  const [saving, setSaving] = useState(false);
  
  const handleSave = async () => {
    setSaving(true);
    await window.api.saveProject(project);
    setSaving(false);
  };
  
  return (
    <button onClick={handleSave} disabled={saving}>
      {saving ? '保存中...' : '保存'}
    </button>
  );
}
```

AI不会说："这里有竞态条件，saving状态未正确处理"（这是代码视角）

AI会说：

```
### 场景1：用户在保存过程中关闭窗口
- 用户操作：我点击"保存"按钮，看到显示"保存中..."，此时我误以为已经保存成功，直接按Alt+F4关闭应用窗口
- 代码路径：ProjectSaveButton.jsx:handleSave:5 → window.api.saveProject → 主进程IPC处理 → 数据库写入
- 断裂点假设：在window.api.saveProject执行期间（await挂起），如果用户关闭窗口，渲染进程被销毁，但主进程的保存操作可能仍在异步执行。此时如果保存依赖渲染进程的回调确认，操作会中断；如果保存不依赖回调，用户不知道保存是否成功，下次打开可能看到半成品数据。
- 验证结果：需确认
  - 需要查看主进程IPC处理逻辑：saveProject是否在主进程有独立事务？窗口关闭是否触发渲染进程cleanup中断IPC回调？

### 场景2：用户在保存时连续快速点击按钮
- 用户操作：我网络比较卡，点击保存后按钮没立即变灰，我以为没点中，又点了一次
- 代码路径：ProjectSaveButton.jsx:handleSave:5（第一次点击）→ handleSave:5（第二次点击，此时saving仍为false）
- 断裂点假设：虽然按钮有disabled={saving}，但React状态更新是异步的。在第一次setSaving(true)生效前，如果用户点击足够快（或主线程被阻塞），第二次点击会触发第二个handleSave，导致重复保存。如果saveProject内部没有幂等性校验，可能产生重复数据或覆盖竞争。
- 验证结果：已证实
  - 证据：React的setState是批量异步更新，在极端情况下（如主进程IPC阻塞渲染进程），disabled状态可能延迟生效
  - 修复Diff：
    ```diff
    + const isSavingRef = useRef(false);
      const handleSave = async () => {
    +   if (isSavingRef.current) return;
    +   isSavingRef.current = true;
        setSaving(true);
        await window.api.saveProject(project);
        setSaving(false);
    +   isSavingRef.current = false;
      };
    ```

### 场景3：用户在保存前修改了项目内容但按钮未感知
- 用户操作：我在文本框里修改了项目名称，但光标还在输入框里，我没触发任何blur事件，直接点击保存
- 代码路径：文本框输入 → project对象状态 → ProjectSaveButton接收project → handleSave发送project
- 断裂点假设：如果project状态是引用类型，且文本框修改的是project.name，由于对象引用未变，React可能未重新渲染ProjectSaveButton。但这不是重点。重点是：如果project对象在传递过程中被浅拷贝，或主进程接收时未做脏检查，保存的可能不是最新内容。不过当前代码看起来project是直接传入的，此场景下代码路径正常。
- 验证结果：已排除
  - 防御机制：project作为prop直接传入，只要父组件状态正确更新，保存的就是最新对象。未发现断裂点。
```

给你的实操建议

1. 分批投喂，但按"用户旅程"而非按文件

不要给AI"整个db目录"或"整个api目录"。而是给AI一段完整的用户操作链涉及的代码：

```
"用户点击'生成视频'按钮后，从React组件到API调用再到数据库记录创建，这条链路上的所有代码如下："
[粘贴3-5个相关文件的核心逻辑]
```

2. 故意不给上下文，测试AI的断裂点敏感度

好的AI应该能在信息不全时，主动说"这里缺少XX信息，用户如果做了YY操作，我无法判断会不会出问题"。如果AI总是强行给出确定结论，说明它在幻觉。

3. 用这个指令做"回归测试设计"

AI产出的每个场景，本质上就是一个测试用例。你可以直接让Trae把这些场景转成Playwright测试脚本：

```
基于以上场景，生成Playwright测试脚本：
- 场景1：测试在saveProject的await期间关闭窗口，数据库是否回滚
- 场景2：测试快速双击保存按钮，是否只产生一条记录
```

一句话总结

预设问题=让AI背答案，使用驱动=让AI做用户。

前者查的是"代码是否符合规范"，后者查的是"用户会不会在这里骂娘"。对于你的产品（面向创作者的AI视频工具），后者才是真金白银的Bug。

你现在可以拿一个你最没把握的模块（比如分镜系统的状态保存，或插件加载流程）试试这个指令，看AI能不能想出你没考虑过的用户作死路径。
